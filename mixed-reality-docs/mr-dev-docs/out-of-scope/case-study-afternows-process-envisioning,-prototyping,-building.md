# <a name="case-study---afternows-process"></a>ケーススタディ-AfterNow のプロセス

[ここ](https://www.afternow.io/)では、お客様と協力して、お客様のアイデアと aspirations を、市場に向けた完全に運用可能な具体的な製品とエクスペリエンスに変えることができます。 しかし、1行のコードを記述する前に、構想と呼ばれるプロセスを使用してブループリントを作成します。

![構想とは](images/whatisenvisioning-640px.png)

## <a name="what-is-envisioning"></a>構想とは

構想とは、製品の設計に使用される ide のプロセスです。 大規模なアイデアを1つまたはいくつかの現実的な概念に合理化することができます。

このプロセスによって、最終的に製品にもたらされる多くの問題とブロッカーが明らかになります。 早期および高速に失敗すると、最終的に大量の時間とコストが節約されます。

このプロセス全体で、グループのアイデアと結果に関するメモを提供します。 完了すると、スクリプトのドキュメントと、"act it out" ステップのビデオが表示されます。これは、提案された製品の動作シミュレーションです。 これらは、お客様の製品の具体的な設計図であり、保持できるフレームワークです。 理想的には、次の手順は製品を構築することです。これにより、実際の処理をテストして体験することができます。

ここでは、プロセスの1日1回の再生、各手順の所要時間、チームで必要とされるユーザーについて説明します。

![受講者の構想](images/envisioning-participants-640px.png)

## <a name="envisioning-participants"></a>受講者の構想

このプロセスに適した参加者を選択することが重要です。 数値は異なる場合がありますが、効率を高めるために、7つのメンバーの下で保持することをお勧めします。

有向 **(重大)** -これは、グループ内の重要な人物です。 転換は、プロセス中に決定が行われた場合 (通常は CEO、創設者、製品マネージャー、または設計の頭)、最終的な決定を行うユーザーか、スケールを調整する人です。 プロセス全体を参加させることができない場合は、ロールに対して最も適したユーザーを指名します。

**Finance (オプション)** -プロジェクトの資金調達方法を説明できるユーザー。

**マーケティング (オプション)** -会社のメッセージを工芸する人。

**お客様 (重要)** -お客様に頻繁に連絡し、その代弁者となる人がいます。

**Tech/ロジスティック (重要)** -企業が構築および提供できる内容を理解している人。

**Design (重要)** -会社が行う製品を設計するユーザー。

## <a name="day-1---map"></a>1日目-マップ

**制約を & の目標を一覧表示する長期的な目標を設定する-5 分**

完全にオプティミスティックである場合、長期的な目標は何ですか。 このソリューションを mixed reality で構築するのはなぜですか。また、解決しようとしている問題は何ですか。

**スプリントの質問の一覧-5 分**

少しの実践的な情報を入手して、問題の形式で懸念を一覧表示します。 どのようにして失敗するのでしょうか。 長期的な目標を達成するには、どうすればよいでしょうか。 プロジェクトが失敗した場合、その原因となる可能性があります。

**フローマップを1時間以内に作成する**

これは大まかな機能マップです (詳細はありません)。 左側にすべてのユーザーの種類を記述し、右端には終わりを記述します。 次に、ユーザーが製品と対話する方法を示すテキストと矢印を中央に接続します。 これは、5-15 の手順で行います。

**インタビュー & の専門家のための注意**

この演習の目的は、問題の総合的な理解を深めることです。 ここでは、マップに項目を追加して、それに対する間違いを修正することができます。 フローの一部をさらに詳細に拡張できる場合は、専門家に問い合わせます。

このプロセスのベストプラクティスは、(HMW) 形式で個々のメモを取得した場合に発生します。 HMW は、70s で P&G によって開発された手法です。 これは、最終的には整理および優先順位付けされたメモを生成する質問の形になります。 付箋で始まる1つのアイデアをメモしておきます。 "どうすればよいか" という語句で始まります。 たとえば、mixed reality で [your company] の値を伝えるにはどうすればよいでしょうか。

![整理と決定](images/organize-and-decide-640px.png)<br>
*整理と決定*

**HMW のメモを整理する**

私たちは、"どのようにして" どのようにして "どのようなものにするか" について説明します。 その後、目立つテーマに分類されます。

**HMW メモに投票する**

すべてのメモが適切なカテゴリに分類されたら、お気に入りの質問に投票します。 全員が2票を取得します。この場合、4票が取得されます。

**ターゲットを選択する**

マップに最も重要な顧客と1つの目標を丸で囲んで、スプリントの最初の質問を確認します。 この呼び出しを実行するタスクがあります。

## <a name="day-2---sketch"></a>日 2-スケッチ

**既存の製品のデモ**

スプリントグループ内のすべてのユーザーは、お客様が取り組んでいるものと同様の問題を解決する既存の製品を調査します。 次に、グループの残りの部分について、結果をデモします。 これにより、興味深い機能がわかります。

**注**

メモを収集するために、スペースをサイレントに調査するのに20分かかります。

**アイデア**

私たちは、約20分の書き留めをプライベートに費やしています。

**8s**

8フレームの作成には8分かかります。 ここでは、各フレームで最も優れたアイデアの1つのバリエーションを迅速にスケッチします。

**解決方法**

ストーリーボードは30分かかります。 ここでは、8からのアイデアを最適なものにして、3つのパネルのストーリーボードを作成します。

![ストーリーボードのスケッチ](images/sketching-storyboards-640px.png)<br>
*ストーリーボードのスケッチ*

## <a name="day-3---decide"></a>日 3-決定

この日には、critiquing と投票が多数含まれています。 次に、その内訳を示します。

**Quiet critiquing**

ここでは、壁にテープソリューションを作成し、それらをサイレントに見てみましょう。 すべてのスケッチをご覧になると、ヒートマップが作成されます。 各ユーザーは、好きなスケッチにマークを付けます。 フロー全体またはフローの領域を指定できます。

**説明 & 説明**

興味のあるアイデアのヒートマップが表示されるようになります。また、注目すべき各スケッチに3分かかります。 特定のスケッチやアーティストが誤解を受けたときに、その理由を正しく伝えられる理由について、ユーザーが明らかにすることができます。 これにより、新しいアイデアが出現する可能性があります。

この手順では、多くの注意を払っていないアイデアを説明する機会を担当者に提供することもできます。 場合によっては、前に理解していなかった gem があると考えられます。

**Straw poll & スーパー投票**

各ユーザーは、自分の好みのアイデアをサイレントに選択し、準備ができたら、お気に入りに1つのマークを一度に配置します。 次に、最も気に入っているアイデアに対して3つの大きなドットが表示されます。 これは、プロトタイプとして考えられるアイデアです。

## <a name="day-4---prototype"></a>日 4-プロトタイプ

**準備**

プロトタイプ作成フェーズの準備として、子の play kit とよく似た一連の物理ツールを収集して、ホログラムを表す3D オブジェクトを構築します。 また、これらのホログラムにアクセスするために使用しているハードウェアデバイスを表す物理ツールも用意されています。

***プロトタイプ!***

**スクリプトを準備する**

このプロトタイプのユーザーフロー全体をスクリプト化し、skit でパーツを演奏するアクターを決定します。

**操作する**

すべてのユーザーパーティが使用する順序で、ユーザーフローを物理的に操作します。 これにより、この製品を構築するときに、どのように実行可能かを把握できます。また、この考え方を理解すると、見過ごされる可能性のあるいくつかの重要なあたっが見られます。
